Scalability
One laptop. 8 AI workers. 22k descriptions.
💻
Distributed across machines
Each team member ran the generation script on their own laptop. Each machine processed a different source and exported a CSV — then all CSVs were merged into the shared database.

⚡
8 Haiku VLM instances simultaneously
Each laptop ran 8 concurrent workers. Every worker fetched a listing image from S3 and sent it in parallel to a separate Claude Haiku Vision call — multiplying throughput 8×.

Semantic Embeddings
Model all-MiniLM-L6-v2 (384-dim vectors, pre-loaded at startup). The LLM writes a 2-sentence ideal listing description from the user's query. Both the ideal and each candidate's combined text are encoded — then ranked by cosine similarity.
Text fields encoded
title description AI image description features
Catches "bright, spacious loft feel" even when those exact words aren't in the listing.

How it all connects
End-to-end user session
1
User ID — cookie
On first visit, getUserId() checks the browser for a rr_user_id cookie. If missing, it generates a random UUID and stores it for 1 year. Every subsequent visit reuses the same ID — no login needed.
rr_user_id = "a3f8…9c1b"
expires: +1 year
2
Interaction tracking
Every action fires silently to POST /users/{id}/interactions
👆 Click a listing click
⏱ Look at listing >N sec view
🖼 Browse images image_browse
3
Profile regeneration
Every 5 new interactions, the backend calls GET /users/{id}/profile. An LLM reads the history and infers preferences. The profile is cached — and injected into every subsequent search.
Same query, different results — because your behaviour changed the weights.
