───────────────────────────────────────────────────────────────────────
  CLAUDE CODE — THE BACKBONE OF OUR WORKFLOW
───────────────────────────────────────────────────────────────────────

We built this entire project inside Claude Code, Anthropic's CLI for
agentic software development. Every participant module, every API route,
every test, every script — written, debugged, and iterated on through
Claude Code sessions.

What made it different from using a chat window:

  • Claude Code reads the actual files. When we asked "why is the
    transit score always zero?", it grepped the codebase, traced the
    call stack, found the bug (a None check in the wrong place), and
    fixed it — without us having to copy-paste a single line.

  • It understands context across the whole repo. When we added the
    image description column, it updated the bootstrap script, the
    parser, the ranking module, and the import script in one go —
    keeping everything consistent.

  • It ran our scripts, watched the output, and self-corrected. When
    the image generation script hit an S3 timeout on the first run,
    Claude Code diagnosed the retry logic, patched it, and re-ran.

  • It handled git for us. Branching, committing with meaningful
    messages, pulling before pushing to avoid overwriting a teammate's
    work — all done autonomously.

We also used the /fewer-permission-prompts skill to streamline our
workflow, and leaned heavily on parallel agent execution to research
the codebase while simultaneously writing code — cutting iteration
time dramatically.


───────────────────────────────────────────────────────────────────────
  THE FRONTEND-DESIGN PLUGIN — A PITCH DECK THAT LOOKS THE PART
───────────────────────────────────────────────────────────────────────

When it came time to build the pitch presentation, we installed the
Claude Code frontend-design plugin — a specialised skill that generates
production-grade, visually distinctive interfaces rather than generic
AI-looking output.

The result: PITCH.html — a fully self-contained, keyboard-navigable
11-slide deck built entirely in HTML/CSS/JS, zero dependencies, that
opens in any browser. It uses the exact RobinReal brand palette
(#1f5f8b navy blue, #cf5c36 accent orange, IBM Plex Sans), features
animated slide transitions, dot navigation, and dark/light slide
combinations designed to hold attention in a live presentation.

Every design decision — the fallback two-column layout, the weight-decay
bar charts, the mock comparison table, the screenshot placeholder grid —
was generated by Claude Code with the frontend-design plugin active,
then refined through conversation until it looked exactly right.

We didn't open Figma. We didn't write a single line of CSS by hand.


───────────────────────────────────────────────────────────────────────
  CLAUDE HAIKU — THE BRAIN OF THE SEARCH PIPELINE
───────────────────────────────────────────────────────────────────────

Model: claude-haiku-4-5-20251001

The search pipeline makes two Claude Haiku calls on every query —
one for hard facts, one for soft preferences. We chose Haiku for its
combination of speed and structured output quality. At datathon scale,
latency matters.

  ┌─ HARD FACT EXTRACTION ──────────────────────────────────────────┐
  │                                                                  │
  │  Haiku receives the raw natural-language query and a structured  │
  │  tool definition. It responds with a validated JSON object       │
  │  containing every non-negotiable constraint it can identify:     │
  │                                                                  │
  │    • City name (normalised to Swiss DB variants)                 │
  │    • Postal code, canton code                                    │
  │    • Price range (understands CHF, Fr., Franken)                 │
  │    • Room range (Swiss half-room convention: 3 rooms → 3.0–3.5)  │
  │    • Living area in m²                                           │
  │    • Availability date (parsed in EN / DE / FR)                  │
  │    • Boolean amenities: balcony, garage, pets, wheelchair…       │
  │    • Offer type: RENT vs SALE                                    │
  │                                                                  │
  │  If Haiku times out (5 s) or raises any exception, the system    │
  │  falls back instantly to a 20+ regex rule engine — zero downtime │
  │  even if Anthropic's servers are unreachable.                    │
  └──────────────────────────────────────────────────────────────────┘

  ┌─ SOFT FACT EXTRACTION ──────────────────────────────────────────┐
  │                                                                  │
  │  A second Haiku call extracts everything qualitative — the       │
  │  things the user wants but won't walk away without:              │
  │                                                                  │
  │    • 18 preference categories (bright, quiet, modern, garden…)   │
  │      each with a numeric weight the model assigns from language  │
  │      cues ("must have" → 1.0, "if possible" → 0.2)              │
  │    • Negative signals: things to explicitly avoid                │
  │    • Target landmark for commute: "ETH Zürich", "Zurich HB"      │
  │    • An ideal listing description — a 2-sentence synthesis of    │
  │      what the perfect property looks like, used as the query     │
  │      vector for semantic embedding similarity                    │
  │                                                                  │
  │  This output feeds directly into the soft scoring and ranking    │
  │  stages, making preferences first-class citizens of the search.  │
  └──────────────────────────────────────────────────────────────────┘

The combination of structured tool-use and a tight system prompt with
Swiss-specific context (189 city name variants, canton codes, half-room
conventions, multilingual date formats) made Haiku punch well above its
weight class on this domain.


───────────────────────────────────────────────────────────────────────
  CLAUDE HAIKU VISION — 22,000 IMAGES DESCRIBED IN PARALLEL
───────────────────────────────────────────────────────────────────────

Model: claude-haiku-4-5-20251001 (vision)

This is the part we're most proud of.

The dataset contained three sources of listing images — SRED (11,105
listings), COMPARIS (10,917 listings), and ROBINREAL (797 listings).
Most search systems would ignore those images entirely. We didn't.

We wrote a generation pipeline that:

  1. Fetches each listing's first available image — from local disk
     (SRED montage files) or directly from S3 URLs (COMPARIS, ROBINREAL)

  2. Base64-encodes the image and sends it to Claude Haiku Vision with
     a carefully crafted system prompt:

       • SRED images are 2×2 montages (top-left = bathroom,
         top-right = interior, bottom-left = kitchen,
         bottom-right = exterior). Haiku describes each quadrant
         in one sentence: lighting, style, condition, features.

       • COMPARIS and ROBINREAL images are single-view photos.
         Haiku writes 2–3 factual sentences covering room type,
         lighting, style, and notable visible features.

  3. Stores the description in a CSV, keyed by listing_id.

  4. The CSV is imported into listings.db — adding an image_description
     column that flows into text matching, embedding similarity, and
     the "What you'll love" section of the detail modal.

The key engineering decision: 8 concurrent workers per machine.

Rather than processing images sequentially, each machine ran a
ThreadPoolExecutor with 8 workers — each worker independently fetching
an image and calling Haiku Vision in parallel. On a single laptop,
this multiplied throughput 8× compared to serial execution.

We distributed the work across team members' machines by source:
one machine handled SRED, another COMPARIS, another ROBINREAL.
Each exported a CSV. We merged them. The whole thing took under
4 minutes per source on commodity hardware.

Total: 22,765 AI-generated image descriptions in the database.

The impact on search quality is concrete: when a user searches for
"bright modern kitchen", listings whose images Haiku described as
"a sleek, renovated kitchen with white cabinetry and abundant natural
light" score higher on both text matching (174 synonym mappings) and
semantic embedding similarity — even if the listing's own text never
used those words.

Images became searchable. That's the win.
