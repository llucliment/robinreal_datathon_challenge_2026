Overview

What is RobinReal?

RobinReal is a natural language real estate search engine for the Swiss market. Instead of clicking through filters, users describe what they want in plain text — in English, German, or French — and the system finds and ranks the best matching listings.

The system combines Large Language Models for intent extraction, semantic embeddings for relevance scoring, real-time Swiss public transport data for proximity ranking, and a user profile engine that personalizes results based on past searches.

The result is a search experience that understands nuance: "bright flat near ETH Zürich for a young professional, max CHF 2'200" is understood as well as a human agent would understand it.

🎯

Competition Objective

Search quality and ranking effectiveness across a SQLite database of Swiss property listings, given a natural-language query.

⚡

Key Differentiators

  • LLM extraction with instant rule-based fallback
  • Real Swiss transit API for commute ranking
  • Multilingual (EN / DE / FR)
  • Semantic embedding similarity
  • Personalization via user profiles

Stage 1 — Hard Fact Extraction

Understanding Strict Requirements

The hard extraction stage identifies non-negotiable constraints: things the user absolutely requires. It runs in two modes with automatic fallback:

🤖

Primary: Claude Haiku with Tool Use

A structured tool call forces the LLM to output a validated JSON object. The system prompt includes Swiss-specific context (room conventions, canton codes, multilingual names). The model understands "3 rooms" → min 3.0, max 3.5 (Swiss half-room convention).

📐

Fallback: Regex Rule Engine

If the LLM call fails or times out (5s), 20+ handcrafted regex patterns extract constraints from the query. Handles price formats (CHF/Fr./Franken), dates in 3 languages, room ranges, area in m², canton codes.

🇨🇭 Swiss-Specific Intelligence

  • 189 city name variants (incl. umlauts, FR/DE pairs)
  • 26 canton codes + full name mapping
  • Half-room convention (1.5, 2.5, 3.5 Zimmer)
  • Swiss price formats (CHF, Fr., Franken)
  • 3-language month names for date parsing

Stage 2 — Hard Filtering

SQL-Powered Candidate Retrieval

Hard filters are translated into parameterized SQL WHERE clauses (no injection risk) against the SQLite listings database. The query retrieves a pool of at least 200 candidates to ensure downstream price statistics are market-representative.

Geographic

  • City (case-insensitive)
  • Postal code list
  • Canton code
  • Radius (haversine km)

Property

  • Price range (CHF)
  • Room range (half-rooms)
  • Living area (m²)
  • Availability date

Amenities

  • Balcony / Elevator
  • Parking / Garage
  • Pets allowed
  • Wheelchair accessible
  • Minergie certified

Classification

  • RENT vs SALE
  • Object category
  • Sort (price/rooms)
  • Limit + offset

🔑 Pool Expansion Strategy

The system always fetches max(limit+offset, 200) listings from the database — even if the user only wants 25 results. This ensures the price z-score normalization in Stage 4 reflects the actual filtered market, not just the first page.

Availability Logic

Listings with NULL availability dates or the sentinel 1970-01-01 are treated as immediately available — a realistic assumption for the Swiss rental market where many listings don't specify a date.

Pagination

Offset is applied after soft filtering and ranking (not at the DB level). This guarantees that page 2 shows the next-best results, not just the next batch from an arbitrary SQL query.

06

Stage 4 — Soft Scoring

Multi-dimensional Candidate Evaluation

Each candidate listing is scored across multiple dimensions using a multi-threaded engine (8 worker pool). Every score is normalized to [0, 1].

💰

Price Value Score

Z-score normalization relative to the filtered candidate pool. A listing priced 1 standard deviation below the mean scores near 1.0; 1 SD above scores near 0.0. Reflects value for the specific market, not absolute price.

🚋

Transit Proximity Score

If a landmark was specified, uses the Swiss transport.opendata.ch API to find the nearest station and compute real transit time. Pre-filters to 40 closest candidates (haversine) to minimize API calls. Cached at 2048 entries.

📝

Text Matching Score

174 synonym mappings across EN/DE/FR for each preference label. Uses substring search + synonym overlap + word-level partial matching against title + description + image descriptions + feature tags.

🏫

Proximity Scores

Linear decay functions for: school (≤800m = 1.0), kindergarten (≤600m), shops (≤500m), public transport (≤1000m). Requires listings with coordinates.

🚫

Negative Signal Penalty

Fraction of negative keywords found in the listing text. Applied as a multiplicative penalty to the final score: score × (1 − penalty).

🌐

Multilingual Synonyms

174 synonym mappings per preference label across English, German, and French ensure listings described in any language score correctly regardless of the query language.

07

Stage 5 — Ranking

The Scoring Formula

The ranking model combines all signals into a single score using a weighted aggregation over stated preferences, semantic similarity, and proximity — plus an optional user profile multiplier.

// 1. Weighted preference aggregation score = Σ(pref.weight × soft_scores[pref.label]) / total_weight // 2. Semantic embedding (if ideal_description) += cosine_sim(ideal_desc, listing_text) × 1.0 // 3. Landmark proximity (if target) += transit_score × 1.0 // 4. Price value tiebreaker += price_z_score × 0.2 // 5. Negative penalty ×= (1.0 − negative_signal_penalty) // 6. User profile multiplier [0.1 → 1.3] ×= profile_multiplier(candidate, user_profile) // 7. Clamp final_score = min(1.0, score)

🧠 Semantic Embeddings

Model: all-MiniLM-L6-v2 (384-dim vectors, pre-loaded at startup). The LLM generates a 2-sentence "ideal listing" description from the user's query. The model encodes both this description and each listing's text (title + description + image text). Cosine similarity between normalized embeddings gives a semantic relevance score independent of keyword matching.

👤 User Profile Personalization

Built from interaction history (clicks, views, image browses). Applies a confidence-weighted multiplier (0.1–1.3×) based on: city affinity, feature affinity, budget fit, aesthetic preferences, negative pattern penalties. New users start with confidence=0 (no boost).

💬 Human-Readable Explanations

Every result includes an auto-generated reason string: "+ bright, quiet | ~ transport | - garden | ~12 min by transit | below-average price" — shown as color-coded badges in the UI.

Notable Technical Decisions

Decision Implementation Why it matters
LLM + Rule Fallback 5-second timeout on Claude; instant regex fallback on any exception Zero downtime if API is slow or rate-limited; search always returns results
Pool Expansion Fetch ≥200 candidates before ranking; apply offset after ranking Price z-scores reflect real market context; page 2 = truly next-best results
Parallel Scoring ThreadPoolExecutor with 8 workers for transit API calls Transit scores for 40 candidates in parallel, not sequentially
Aggressive Caching LRU cache: 64 geocode, 512 station lookup, 2048 transit time Repeated queries for same landmarks are instant
Embedding Pre-load Model loaded at server startup; warm-up call in FastAPI lifespan First search is fast; no cold-start delay on first query
Single-URL Deployment Frontend built into Docker image; served via FastAPI StaticFiles One URL for the jury — no CORS issues, no separate hosting
Multilingual City Map 189 hand-curated city → DB variant mappings User typing "Zurich", "Zürich", "Zuerich" all hit the same listings
Parameterized SQL All DB queries use placeholders, never string interpolation No SQL injection risk regardless of user input
